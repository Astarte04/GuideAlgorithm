# 中国大陆停车场收费与余位数据获取调研文档

> 调研日期：2026-05-07  
> 目标：为中国大陆客户获取周边停车场的收费信息、总车位、实时/准实时余位、营业时间、经纬度、入口、预约能力等数据，评估可落地的数据来源与技术路线。

## 1. 结论摘要

中国大陆停车数据没有一个全国统一、公开、稳定的免费接口。要同时拿到“停车场位置 + 收费规则 + 总车位 + 实时余位”，建议采用多源融合方案：

1. **第一优先级：城市级智慧停车/公共数据开放平台**。这类来源最合规、余位可信度最高，但城市之间接口和开放程度差异很大。
2. **第二优先级：商业停车数据 API / 停车平台商务合作**。适合快速覆盖多个城市，可获取总车位、剩余车位、费用描述等，但需要付费和签约。
3. **第三优先级：地图 POI API**。高德、百度、腾讯位置服务适合补全停车场名称、地址、经纬度、POI 分类、电话、营业状态等基础信息，但公开开发者 API 通常不承诺返回实时余位和完整收费规则。
4. **第四优先级：停车场运营商/设备厂商直连**。对医院、商圈、景区、园区等重点场景最准确，可通过道闸、车牌识别、诱导屏、停车云平台获取实时余位和收费规则。
5. **第五优先级：合规爬虫/信息抽取**。只建议用于补充收费公告、政府定价文件、停车场官网/小程序公开页面，不建议绕过登录、加密签名、验证码或反爬机制抓取 App 私有接口。
6. **兜底方案：视觉识别/传感器/预测模型**。当没有数据源时，可通过摄像头、地磁、道闸计数或历史预测构建自有余位能力，但建设成本更高。

推荐落地路径：**先选目标城市做试点，城市平台 + 聚合 API + 地图 POI 建基础库；重点停车场通过商务直连；最后用合规爬虫和人工校验补齐收费规则。**

## 2. 目标数据字段

建议将所有数据源统一映射为如下模型：

| 字段 | 含义 | 来源优先级 |
| --- | --- | --- |
| `source` | 数据来源标识 | 系统生成 |
| `source_parking_id` | 来源侧停车场 ID | 城市平台/API/运营商 |
| `name` | 停车场名称 | 地图 POI、城市平台、运营商 |
| `address` | 地址 | 地图 POI、城市平台 |
| `longitude` / `latitude` | 经纬度 | 地图 POI、城市平台 |
| `coord_type` | 坐标系：GCJ-02、BD-09、WGS-84 | 数据源说明 |
| `entrances` | 入口坐标列表 | 地图详情、运营商、人工维护 |
| `total_spaces` | 总泊位数 | 城市平台、运营商、商业 API |
| `open_spaces` | 当前开放泊位数 | 城市平台、运营商 |
| `available_spaces` | 剩余泊位数 | 城市平台、运营商、商业 API |
| `available_status` | 充足/紧张/满位/未知 | 系统计算或来源返回 |
| `fee_rule_text` | 原始收费描述 | 商业 API、政府文件、爬虫、人工维护 |
| `fee_rule_structured` | 结构化收费规则 | 信息抽取/人工校验 |
| `opening_hours` | 开放时间 | 城市平台、地图 POI、运营商 |
| `parking_type` | 商业、医院、景区、路内、P+R 等 | 城市平台、地图 POI |
| `can_reserve` | 是否支持预约 | 商业 API、运营商 |
| `updated_at` | 数据更新时间 | 来源返回/采集时间 |
| `confidence` | 可信度评分 | 系统计算 |

## 3. 可获取路径一：政府开放数据与城市智慧停车平台

### 3.1 北京公共数据开放平台

北京公共数据开放平台存在多个与停车相关的数据集：

| 数据集 | 可用字段 | 获取方式 | 备注 |
| --- | --- | --- | --- |
| 实时停车泊位 | 停车场 ID、停车场编号、停车场名称、开放泊位数、剩余泊位数、空闲类型 | CSV/XLSX 下载、平台 API | 平台页面展示接口形态为 `https://data.beijing.gov.cn/cms/web/api/{userKey}/{contentId}`，需要登录获取 `userKey`。 |
| 停车场设施信息 | 车场编号、车场名称、泊位数量、所属单位、所在道路、收费方式 | CSV/XLSX 下载、平台 API | 适合建设北京静态车场库。 |
| 路侧停车位基础信息 | 停车场名、道路位置、泊位数量、经度、纬度 | CSV/XLSX 下载、平台 API | 适合路侧泊位基础库。 |
| 停车场收费标准清单信息 | 停车场类别、车辆类别、免费时间、计次/计时收费、所属区县、备注 | CSV/XLSX 下载、平台 API | 更偏政策/分类收费清单，不一定是每个具体停车场的实时价格。 |

参考来源：

- 北京“实时停车泊位”数据集说明：<https://data.beijing.gov.cn/zyml/wnkfsj/c5db32fdc4584b7fbad9c87af1276395.htm>
- 北京“停车场设施信息”数据集说明：<https://data.beijing.gov.cn/zyml/AIsj/4bc1b7356a2c4a309335949c4a4ee8db.htm>
- 北京“路侧停车位基础信息”数据集说明：<https://data.beijing.gov.cn/zyml/ajg/sjtw/17457.htm>
- 北京“停车场收费标准清单信息”数据集说明：<https://data.beijing.gov.cn/zyml/AIsj/da48e160616749ac82af8f3fa4a4cd52.htm>

可行性判断：**高**。北京是比较适合作为验证城市的样板，但要确认数据更新频率是否满足业务实时性要求。

### 3.2 深圳公共智慧停车平台

深圳政府在线披露，深圳公共智慧停车平台整合发布了经营性停车场的位置、车位总数、空余车位、收费信息、开放时段等动静态信息，并面向市民提供数据质量评价功能。

参考来源：<https://www.sz.gov.cn/cn/xxgk/zfxxgj/bmdt/content/post_11632741.html>

可行性判断：**高，但需要进一步商务/政务对接**。公开报道表明数据存在且集中到平台，但未直接发现面向第三方开发者的公开 API 文档。建议以“政务数据合作/应用接入/停车诱导服务”方式联系深圳交通运输部门或平台运营方。

### 3.3 上海停车公共服务体系

上海市交通委员会对“上海停车”APP 的问答中说明，该 APP 覆盖经营性停车设施、收费道路停车场、P+R、交通枢纽等，提供目的地周边停车场地址、出入口位置、实时空泊位数、收费价格、服务时间等信息。

参考来源：<https://jtw.sh.gov.cn/zsk/20211206/b2a2c4a46e3b4e3e9cc1820f5a8db4a1.html>

可行性判断：**中高**。数据能力明确存在，但是否能对第三方开放，需要通过上海交通委/上海停车平台或数据开放平台申请。若只通过 App 页面或小程序抓取，合规与稳定性风险较高。

### 3.4 广州泊车与统一停车信息平台

广州市政府门户网站披露，广州建设全市统一停车信息平台，并通过“广州泊车”小程序向市民开放试用，目标包括停车资源实时更新、查询、预订与导航服务一体化。

参考来源：<https://www.gz.gov.cn/zwfw/zxfw/gysy/content/post_7201780.html>

可行性判断：**中高**。建议优先尝试政府/平台合作；同时可从政府公开计划清单、区级经营性停车场基础信息入手建立静态库。

### 3.5 其他城市的排查方法

对每个目标城市，建议按以下关键词检索：

```text
城市名 + 停车场 + 剩余泊位 + 开放数据
城市名 + 智慧停车 + API
城市名 + 公共停车 + 实时空位
城市名 + 道路停车 + 收费标准
城市名 + 政府定价 + 停车场收费标准
城市名 + 停车小程序 + 空余车位
```

常见入口包括：

- 市级公共数据开放平台。
- 市交通运输局/住建局/城管局/发改委网站。
- 城市智慧停车 APP/小程序，如“上海停车”“广州泊车”“宜停车”等。
- 政务服务网、政府价格收费目录、停车收费标准 PDF。

## 4. 可获取路径二：商业停车数据 API

### 4.1 聚合数据：全国停车场实时数据查询

聚合数据提供“全国停车场实时数据查询”API，页面说明可根据经纬度查询全国 60 个城市、10 万个停车场实时数据，包含停车场位置、车位数、剩余车位、费用等信息。

参考来源：<https://www.juhe.cn/docs/api/id/366>

可行性判断：**高，适合快速验证**。需要实名认证、申请 API、确认授权范围、调用限制、城市覆盖、数据更新频率、商业使用价格。

### 4.2 极速数据：全国停车场实时数据查询

极速数据页面说明其停车接口覆盖 69 个城市、12 万个停车场，包含位置、车位数、剩余车位、费用等信息；动态数据约 20 分钟更新一次，静态数据每月更新一次。

参考来源：<https://www.jisuapi.com/api/parking/> 与移动页 <https://m.jisuapi.com/api/parking/>

可行性判断：**高，适合作为聚合数据的替代或备份源**。需要重点验证城市清单、余位准确率、字段完整性、商用许可和 SLA。

### 4.3 ThinkAPI / TopThink 停车接口

ThinkAPI 文档中展示了“查询指定城市停车场信息列表”“查询周边停车场信息列表”“获取停车场详情信息”“获取开放停车场查询的城市列表”等接口，并展示字段：`total`、`leftNum`、`price`、`priceUnit`、`priceDesc`、`isDestine`、`reservePrice` 等。

参考来源：<https://www.kancloud.cn/topthink-doc/think-api/2053891>

可行性判断：**中高**。接口字段贴近需求，但需要确认数据源授权、当前维护状态、计费方式和可商用条款。

### 4.4 六派数据等 API 集市

六派数据也存在“全国停车场实时数据 API”页面，宣称提供停车场位置、车位、剩余车位、费用等能力。

参考来源：<https://www.6api.net/api/parking/>

可行性判断：**中**。API 集市类服务要重点做交叉验证：同一停车场与官方平台/实地数据比对，确认是否只是转售同一上游。

### 4.5 停车平台/设备厂商商务合作

国内智慧停车平台和设备厂商通常掌握真实停车场数据，适合通过商务合作获取。可重点关注：

| 平台/厂商类型 | 可能数据 | 获取方式 |
| --- | --- | --- |
| 停简单 | 场库、余位、支付、订单、导航 | 商务合作，App Store 页面显示覆盖城市和场库规模。参考：<https://apps.apple.com/cn/app/%E5%81%9C%E7%AE%80%E5%8D%95/id987821273> |
| ETCP | 停车价格、空闲状态、空余车位实时更新 | 商务合作。参考：<https://www.etcp.org.cn/owners.html> |
| 捷停车/捷顺科技 | 停车缴费、场库运营、设备平台 | 商务合作/生态接入。参考：<https://www.jieshun.cn/jparking> |
| 道闸/车牌识别厂商 | 进出场记录、剩余车位、收费规则 | 项目直连/本地部署/云平台 API |

可行性判断：**高，但商务成本高**。如果客户有重点目的地资源，如医院、商场、景区、机场、园区，运营商直连的准确性通常优于聚合 API。

## 5. 可获取路径三：地图 POI API

地图 POI API 适合解决“附近有哪些停车场、在哪里、怎么去”的问题，但不能作为实时余位的主数据源。

### 5.1 高德地图 Web 服务 API

高德搜索服务 API 提供关键字搜索、周边搜索、多边形搜索、ID 查询等 POI 检索机制，可用关键字“停车场”或停车场类型码建立基础 POI 库。

参考来源：<https://amap.apifox.cn/doc-539873>

建议用途：

- 以经纬度和半径做周边停车场搜索。
- 获取停车场名称、地址、经纬度、POI 类型、行政区划等。
- 与城市/商业 API 做名称、地址、距离匹配，解决数据去重。

限制：公开搜索 POI API 主要返回静态地点数据，通常不保证返回实时剩余车位和完整收费规则。若高德 App 端可见停车余位，不代表开放平台 API 一定可取用。

### 5.2 百度地图 Place API

百度地点检索服务支持城市检索、圆形区域检索、多边形区域检索和地点详情检索，圆形区域检索适用于周边检索场景，地点详情可获取评分、营业时间等类别详情。

参考来源：<https://lbsyun.baidu.com/index.php?title=webapi%2Fguide%2Fwebservice-placeapi>

建议用途：

- 作为高德 POI 的备份源。
- 用于百度坐标系生态下的周边停车场检索。
- 获取 POI 标签，如地上/地下停车场标签、营业状态等。

限制：同样不应预期其公开 Place API 返回实时余位。

### 5.3 腾讯位置服务

腾讯位置服务可作为地图展示、地点搜索、导航组件、小程序生态的补充来源。

参考来源：<https://cloud.tencent.cn/solution/lbs>

建议用途：

- 若产品部署在微信小程序生态，可优先评估腾讯位置服务。
- 结合城市停车小程序入口做用户体验闭环，但数据获取仍需合规授权。

## 6. 可获取路径四：GitHub/Open Source 能提供什么

本次在 GitHub 上按“停车场 剩余车位 API 爬虫”等关键词检索，未发现可直接合法获取中国大陆城市停车余位/收费的通用开源爬虫或 API 项目。GitHub 对该关键词组合显示仓库结果为 0。参考搜索：<https://github.com/search?q=%E5%81%9C%E8%BD%A6%E5%9C%BA+%E5%89%A9%E4%BD%99%E8%BD%A6%E4%BD%8D+API+%E7%88%AC%E8%99%AB&type=repositories>

GitHub 更有价值的方向是：

1. **自建停车位识别/余位检测**：如基于 YOLOv5/YOLOv8、OpenCV、Flask 的停车位检测项目。参考搜索结果：<https://github.com/search?q=%E5%81%9C%E8%BD%A6%E5%9C%BA+%E8%BD%A6%E4%BD%8D%E6%A3%80%E6%B5%8B+yolo&type=repositories>
2. **停车场管理系统模板**：用于理解数据模型、预约、订单、车位状态设计，但通常不是外部数据采集源。
3. **国外停车开放数据示例**：可参考架构和字段设计，但中国大陆数据源仍需本地适配。

结论：**不要指望 GitHub 直接给到中国大陆停车余位数据源；可把 GitHub 用作算法、后台、可视化和数据模型参考。**

## 7. 可获取路径五：合规爬虫与信息抽取

### 7.1 适合爬取/抽取的数据

| 数据类型 | 示例来源 | 价值 |
| --- | --- | --- |
| 政府停车收费标准 PDF/网页 | 发改委、交通局、景区官网 | 获取政府定价、免费时长、封顶价 |
| 停车场官网/商场官网 | 商场、医院、景区 | 获取收费规则、开放时间、入口说明 |
| 城市停车平台公开网页 | 不需要登录且允许访问的页面 | 补充地址、价格、服务时间 |
| 地图 POI 详情页公开信息 | 遵守平台服务条款 | 补充电话、评论、营业信息 |

### 7.2 不建议爬取的数据

- 需要登录、Token、签名、验证码、设备指纹的 App/小程序私有接口。
- 需要逆向 App、破解加密、绕过访问控制的数据。
- 包含车牌、手机号、订单、支付记录、停车轨迹等个人信息的数据。
- 明确禁止爬取或限制商业使用的网站数据。

### 7.3 收费规则抽取建议

收费规则经常是非结构化文本，可采用 LLM + 规则引擎抽取为结构化字段：

```json
{
  "free_minutes": 15,
  "first_period": {"duration_minutes": 60, "price_yuan": 10},
  "next_period": {"duration_minutes": 30, "price_yuan": 5},
  "daily_cap_yuan": 80,
  "night_rule": "22:00-07:00 5元/次",
  "vehicle_type": "小型车",
  "source_text": "15分钟内免费，首小时10元，之后每半小时5元，24小时封顶80元"
}
```

建议所有自动抽取的收费规则都标记 `needs_review=true`，上线前对高流量停车场做人工校验。

## 8. 运营商/设备厂商直连方案

### 8.1 数据接入方式

停车场侧通常已有以下系统：

- 车牌识别摄像机。
- 道闸/出入口控制器。
- 收费管理系统。
- 诱导屏/车位引导系统。
- 云停车平台。
- 电子发票/无感支付系统。

常见接入模式：

| 模式 | 说明 | 适合场景 |
| --- | --- | --- |
| REST API | 停车平台提供查询接口 | 云平台成熟的运营商 |
| WebSocket/MQTT | 实时推送余位变化 | 高实时要求场景 |
| SFTP/CSV | 定时同步静态收费规则 | 收费规则、基础库 |
| 数据库视图 | 专线或内网读取 | 政企/园区本地项目 |
| 边缘网关 | 在停车场侧部署采集网关 | 老旧停车系统改造 |

### 8.2 最小对接接口

停车场实时余位接口建议最少包含：

```http
GET /parking-lots/{parking_id}/availability
```

```json
{
  "parking_id": "P12345",
  "total_spaces": 300,
  "open_spaces": 280,
  "available_spaces": 42,
  "reserved_spaces": 10,
  "monthly_spaces": 80,
  "updated_at": "2026-05-07T10:30:00+08:00"
}
```

收费规则接口建议最少包含：

```http
GET /parking-lots/{parking_id}/fee-rule
```

```json
{
  "parking_id": "P12345",
  "vehicle_type": "small_car",
  "free_minutes": 15,
  "rule_text": "15分钟内免费，首小时10元，之后每半小时5元，24小时封顶80元",
  "effective_from": "2026-01-01",
  "updated_at": "2026-05-07T10:30:00+08:00"
}
```

## 9. 数据融合与质量控制

### 9.1 多源融合策略

同一停车场可能在不同来源中名称略有不同，需要做实体匹配：

1. 坐标距离小于 100 米。
2. 名称相似度高，如去除“地下停车场”“停车库”“P”等后比对。
3. 地址包含同一道路/门牌/商场名称。
4. 电话或入口位置一致。
5. 人工确认重点停车场。

建议按可信度优先级覆盖字段：

```text
实时余位：运营商直连 > 城市平台 > 商业 API > 用户反馈/预测
收费规则：运营商直连 > 政府收费文件 > 城市平台 > 商业 API > 官网/爬虫 > 用户反馈
基础 POI：城市平台 > 高德/百度/腾讯 > OSM > 商业 API > 人工维护
```

### 9.2 可信度评分

建议为每条数据计算 `confidence`：

| 等级 | 来源 | 展示方式 |
| --- | --- | --- |
| A+ | 停车场运营商实时直连，1-5 分钟内更新 | 展示具体余位 |
| A | 城市官方实时数据，5-20 分钟内更新 | 展示具体余位 |
| B | 商业 API，20-60 分钟内更新 | 展示具体余位并标注更新时间 |
| C | 静态收费/总车位数据 | 展示收费、总车位，不展示实时余位 |
| D | 爬虫/用户反馈/预测 | 展示“可能充足/紧张”，避免精确承诺 |

### 9.3 异常检测

- `available_spaces > total_spaces`：丢弃或降权。
- 连续多小时余位不变：标记疑似非实时。
- 高峰时段长期满位/长期空位：与历史分布比对。
- 坐标偏离行政区：触发坐标系或地理编码校验。
- 收费规则缺少生效日期：标记需人工复核。

## 10. 法务与合规注意事项

1. **优先使用授权数据**：政府开放数据、正式 API、商务合作优于爬虫。
2. **遵守平台条款**：地图 API、商业 API、政府开放平台均可能限制缓存、再分发、展示归属、调用频率和商用范围。
3. **不采集个人信息**：车牌号、手机号、订单、支付、停车轨迹属于高风险数据，若业务不必要不要采集。
4. **避免逆向和绕过访问控制**：不要破解 App 签名、验证码、Token、加密参数或私有协议。
5. **标注更新时间和来源**：余位和收费均会变化，前端应展示“更新时间”和“以现场公示为准”。
6. **收费信息需复核**：自动抽取的收费规则必须保留来源截图/链接和采集时间，重要停车场建议人工确认。

## 11. 推荐实施计划

### 第 1 阶段：两周验证

- 选择 1-2 个客户目标城市，例如北京、深圳、上海、广州中的一个。
- 申请并试用聚合数据/极速数据/ThinkAPI 至少两个商业接口。
- 申请高德或百度地图 Web 服务 Key，建立周边停车场基础 POI 查询。
- 对 50-100 个停车场做官方/商业/地图三方匹配。
- 实地或人工抽样验证 20 个停车场的余位和收费准确性。

### 第 2 阶段：一个月 MVP

- 建立停车场主数据表、余位快照表、收费规则表、来源表。
- 接入一个商业 API + 一个地图 POI API + 一个城市官方数据源。
- 做多源去重、坐标纠偏、收费文本结构化、可信度评分。
- 前端展示附近停车场、距离、余位、收费、更新时间、导航入口。
- 对“未知余位”的停车场只展示静态信息，不承诺实时空位。

### 第 3 阶段：商业化扩展

- 与重点城市智慧停车平台或停车运营商谈数据合作。
- 与医院、商场、景区、园区停车场签直接接入协议。
- 建立数据质量监控、异常告警、人工审核后台。
- 在缺少实时数据区域引入预测模型，输出“空位概率”。

## 14. 最终建议

如果客户希望尽快上线中国大陆停车场收费与余位能力，建议不要单押爬虫，也不要只依赖地图 POI。最稳妥的组合是：

```text
城市官方数据 / 智慧停车平台
+ 商业停车实时 API
+ 高德/百度/腾讯 POI 基础库
+ 重点停车场运营商直连
+ 合规爬虫补收费公告
+ 人工抽样校验和可信度评分
```

近期最值得马上做的事情：

1. 明确首批目标城市和客户场景。
2. 申请聚合数据、极速数据或 ThinkAPI 试用，验证字段和准确率。
3. 申请高德/百度地图 Key，搭建 POI 基础库。
4. 在目标城市公共数据平台检索“停车/泊位/收费/余位”。
5. 找 20 个高频停车场做人工/实地比对，形成供应商准确率报告。
6. 与客户掌握资源的停车场或物业谈直连接入。
